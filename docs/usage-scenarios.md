# Work Harvest 사용 시나리오

이 문서는 세션, 브랜치, 저장소 또는 작업 형태가 달라도 같은 기준으로 기록하기 위한 실전 예시다. Work Harvest의 분류 중심은 세션이나 브랜치가 아니라 **보고 가능한 업무 목표와 결과물**이다.

## 빠른 판단표

| 상황 | 업무 항목 | 체크포인트 | 핵심 근거 |
| --- | --- | --- | --- |
| 같은 목표를 여러 세션에서 작업 | 하나를 유지 | 세션·날짜별 누적 | 코드, 문서, 검증 |
| 같은 목표를 여러 브랜치에서 작업 | 하나를 유지 | 브랜치별 또는 단계별 누적 | 브랜치, 커밋, PR |
| 한 세션에서 독립된 결과물 여러 개 작업 | 결과물별 분리 | 각 업무 항목에 기록 | 각 결과물의 근거 |
| 문서·기획·회의·운영 작업 | 독립 업무 항목 가능 | `progress` 또는 `final` | 문서, URL, 결정, 리뷰 |
| 작업 후 별도 세션에서 설명 | 기존 항목 재사용 또는 생성 | `backfill` | 사용자 설명과 확인 가능한 자료 |
| 날짜를 정확히 모르는 과거 작업 | 하나를 유지 | 범위 또는 미상 정밀도 | Git, 문서, 사용자 지정 날짜 |

## 시나리오 1: 긴 Codex 세션에서 며칠간 작업

인증 테스트 완성이라는 하나의 결과물을 같은 세션에서 사흘 동안 작업한다.

매일 작업이 끝날 때 요청한다.

```text
오늘 작업은 여기까지 기록해줘. 이 세션은 내일 계속 사용할 거야.
```

처리 방식:

- 업무 항목 `AUTH-142`를 계속 사용한다.
- 날짜별 체크포인트를 추가한다.
- 세션이 유지되더라도 매일의 작업 경계는 보존한다.
- 성과 노트에서는 여러 체크포인트를 하나의 인증 테스트 업무로 합친다.

## 시나리오 2: 브랜치마다 세션을 나누지만 목표는 동일

한 업무를 `feat/auth-api`, `test/auth-api`, `docs/auth-api` 브랜치와 별도 Codex 세션에서 진행한다.

```text
이 브랜치에서 진행한 인증 API 문서 작업을 기존 AUTH-142 업무에 기록해줘.
```

처리 방식:

- 세 브랜치가 같은 보고 목표를 향하면 `AUTH-142`를 유지한다.
- 각 체크포인트의 Git 정보와 근거에 해당 브랜치·커밋·파일을 기록한다.
- 브랜치가 다르다는 이유만으로 업무 항목을 새로 만들지 않는다.

반대로 문서 작업이 별도 정책 문서처럼 독립적으로 보고할 결과물이라면 새로운 업무 항목으로 분리한다.

## 시나리오 3: 한 세션에서 서로 다른 업무 수행

하나의 Codex 세션에서 인증 버그 수정 후 배포 가이드도 작성했다.

```text
인증 버그 수정과 배포 가이드 작성은 별도 업무로 나눠서 기록해줘.
```

처리 방식:

- 세션이 하나여도 결과물이 독립적이면 업무 항목을 두 개 사용한다.
- 각 업무 항목에 해당하는 활동·검증·결과만 기록한다.
- 한 체크포인트에 서로 무관한 성과를 합치지 않는다.

## 시나리오 4: 코드 변경 없는 문서 작업

Codex에서 운영 정책, 설계 문서 또는 체크리스트를 작성했다.

```text
오늘 작성한 장애 대응 정책 문서를 완료 업무로 기록해줘.
```

처리 방식:

- `work_types`에 `documentation`, `planning` 또는 `operation`을 사용한다.
- 저장소와 브랜치는 없어도 된다.
- 문서 경로, URL, 검토 결과와 확정된 정책을 근거로 남긴다.
- 실제 검토하지 않은 문서를 검증 완료로 기록하지 않는다.

## 시나리오 5: 회의·조율·운영 같은 비코드 업무

회의에서 정책을 확정하거나 운영 장애를 대응했지만 코드 변경은 없었다.

```text
오늘 결제 장애 대응과 재발 방지 결정 사항을 업무 기록으로 남겨줘.
```

처리 방식:

- `operation`, `communication`, `planning` 등의 작업 유형을 사용한다.
- 작업 유형 값은 스키마에 정의된 단수형 `operation`을 사용하며 `operations`는 사용하지 않는다.
- 수행한 대응, 확정된 결정과 이유, 확인한 운영 결과를 기록한다.
- 회의록, 대시보드, 이슈, 메시지 또는 관련 문서를 근거로 연결한다.
- 참석이나 대화 자체보다 보고 가능한 결정과 결과를 중심으로 기록한다.

## 시나리오 6: 작업 후 기록 전용 세션에서 설명

사용자가 지난주에 다른 도구나 오프라인에서 한 작업을 새 Codex 세션에서 설명한다.

```text
지난주 월요일부터 수요일까지 배포 가이드를 정리했어. 지금 말한 내용으로 기록해줘. 문서 링크는 아직 없어.
```

처리 방식:

- 체크포인트 종류는 `backfill`을 사용한다.
- `captured_at`은 지금 기록한 시각으로 둔다.
- 실제 작업 시점은 `work_period`에 월요일부터 수요일까지의 범위로 기록한다.
- 날짜 근거는 `user`, 정밀도는 `range`로 둔다.
- 사용자 설명은 기록할 수 있지만 독립 검증된 사실처럼 표현하지 않는다.
- 문서 확인이 필요하면 검증과 후속 작업에 명시한다.

## 시나리오 7: 외부 문서나 기존 노트로 사후 기록

Notion 문서, 회의록, PR 또는 이슈를 제공하면서 업무 기록 생성을 요청한다.

```text
이 회의록과 PR을 근거로 지난달 마이그레이션 작업을 기록해줘.
```

처리 방식:

- `backfill`로 기록한다.
- 확인한 문서·PR·이슈를 근거에 연결한다.
- 문서에서 직접 확인한 사실과 사용자가 추가로 설명한 내용을 구분한다.
- 제공 자료에 없는 성과나 날짜를 추정하지 않는다.

## 시나리오 8: 정확한 작업 날짜를 모름

작업은 기억하지만 정확한 날짜를 확인할 수 없다.

```text
지난달 중순쯤 진행한 캐시 정책 검토를 기록해줘. 정확한 날짜는 기억나지 않아.
```

처리 방식:

- 확인 가능한 범위가 있으면 `precision: range`를 사용한다.
- 범위조차 신뢰할 수 없으면 `precision: unknown`과 `start/end: null`을 사용한다.
- 커밋 날짜 하나만으로 모든 작업 날짜를 단정하지 않는다.
- 보고서에서도 정확한 일자 대신 확인 가능한 기간으로 집계한다.

## 시나리오 9: 새 세션으로 인수인계

현재 세션의 결과물은 미완료지만 맥락을 새 세션으로 옮기려 한다.

```text
현재 단계까지 기록하고 새 세션에서 이어갈 수 있도록 인수인계를 갱신해줘.
```

처리 방식:

- 먼저 `progress` 체크포인트를 생성한다.
- 현재 상태, 결정, 주요 파일, 완료·미완료 검증, 다음 작업과 리스크를 context에 반영한다.
- 새 세션에서는 전체 대화 대신 업무 항목 context와 실제 코드·문서를 다시 확인한다.

## 시나리오 10: 성과 노트 생성

업무 항목에 체크포인트가 충분히 쌓인 뒤 요청한다.

```text
AUTH-142 기록을 기반으로 성과 노트 초안을 만들어줘.
```

CLI에서는 다음과 같이 생성한다.

```bash
pnpm wh report performance-note --work-item AUTH-142
```

체크포인트에서 확인된 활동·결정·검증·결과는 자동 반영하고, 근거가 없는 정량 성과·배포 결과·사용자 영향은 `미확인`으로 남긴다. 생성 후 작업 규모에 맞지 않는 섹션은 삭제할 수 있다.

## 기록할 때 지켜야 할 공통 기준

- 세션과 브랜치는 근거이지 업무 항목의 절대 식별자가 아니다.
- 코드 변경이 없어도 보고할 목표와 결과가 있으면 기록할 수 있다.
- 사용자의 설명은 유효한 기록 출처지만 독립 검증과 구분한다.
- `captured_at`과 실제 `work_period`를 구분한다.
- 실행하지 않은 검증과 측정하지 않은 성과를 만들어내지 않는다.
- 여러 결과물을 한 기록에 합치거나 같은 결과물을 세션마다 중복 생성하지 않는다.
