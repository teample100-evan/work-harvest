# Work Harness 작업 방법론

- 상태: 확정(일반 운영 원칙)
- 버전: 1.0
- 확정일: 2026-07-13
- 적용 대상: Codex, Claude Code 및 이후 추가되는 로컬 작업 에이전트

## 1. 목적

Work Harness는 에이전트와 함께 수행한 업무를 검증 가능한 기록으로 남기고, 그 기록을 기능·업무 단위로 집계하여 주간 업무 보고와 업무 성과 작성에 활용한다.

이 문서는 다음 질문에 대한 공통 운영 기준을 정의한다.

- 언제 기존 세션을 계속 사용하고 언제 새 세션을 시작하는가?
- 긴 세션의 작업을 어떤 시점에 기록하는가?
- 같은 업무가 여러 세션과 날짜에 걸쳐 진행될 때 어떻게 연결하는가?
- 에이전트가 날짜를 정확히 알 수 없는 경우 어떻게 처리하는가?
- 새 세션이 이전 작업의 맥락을 어떻게 이어받는가?

구체적인 보고서 출력 형식은 별도 템플릿에서 관리한다.

## 2. 핵심 원칙

1. **세션, 업무 항목, 체크포인트, 보고 기간은 서로 다른 단위다.**
2. **세션은 날짜나 대화 길이가 아니라 하나의 뚜렷한 결과물을 기준으로 나눈다.**
3. **같은 결과물을 향한 미완료 작업은 여러 날에 걸쳐 같은 세션에서 계속할 수 있다.**
4. **세션을 계속하더라도 업무일 종료 시 날짜별 체크포인트를 남긴다.**
5. **같은 기능이나 프로젝트라도 결과물이 달라지면 새 세션을 시작한다.**
6. **새 세션은 이전 대화 전체가 아니라 업무 항목의 인수인계 문서와 현재 코드를 읽고 시작한다.**
7. **과거 체크포인트는 수정하지 않고 누적하며, 인수인계 문서는 현재 상태로 갱신한다.**
8. **날짜 근거가 없으면 에이전트가 임의로 일별 작업을 만들어내지 않는다.**
9. **보고서는 세션이 아니라 업무 항목을 기준으로 기록을 합친다.**
10. **활동과 결과, 결과와 업무 성과를 구분하고 모든 성과 문장에 근거를 연결한다.**

## 3. 작업 단위

| 단위 | 의미 | 분리 기준 | 일반적인 지속 기간 |
| --- | --- | --- | --- |
| 프로젝트 | 같은 제품 또는 코드베이스 | 저장소·제품 경계 | 수개월 이상 |
| 업무 항목 | 기능 또는 보고 가능한 업무 단위 | 업무 목적·성과 경계 | 여러 세션·여러 주 |
| 세션 | 하나의 뚜렷한 결과물을 만드는 에이전트 작업 맥락 | 결과물·완료 조건 경계 | 수시간~수일 |
| 체크포인트 | 마지막 기록 이후의 진행 내용을 남기는 불변 기록 | 업무일 종료·단계 완료·세션 전환 | 수시간~하루 |
| 보고 기간 | 기록을 집계하는 달력상의 범위 | 주·월·분기·반기 | 고정 기간 |

### 3.1 관계

- 하나의 프로젝트는 여러 업무 항목을 가진다.
- 하나의 업무 항목은 여러 세션에 걸쳐 진행될 수 있다.
- 하나의 세션은 여러 날짜와 체크포인트를 포함할 수 있다.
- 하나의 체크포인트는 원칙적으로 하나의 업무 항목에 속한다.
- 보고서는 지정한 기간의 체크포인트를 업무 항목별로 집계한다.

## 4. 업무 항목

업무 항목은 Work Harness에서 기록을 분류하고 보고서를 생성하는 중심 단위다. `work_item_id`는 세션이 바뀌어도 유지되는 안정적인 식별자여야 한다.

예시:

```text
프로젝트: jajak-front
업무 항목: AUTH-142 인증 시스템 개선

세션 1: refresh token 구조 설계 및 구현
세션 2: 인증 테스트 코드 작성
세션 3: 운영 환경 오류 대응
```

업무 항목은 최소한 다음 정보를 가진다.

- 식별자와 제목
- 업무 목적
- 현재 상태
- 주요 결정과 제약
- 관련 저장소·이슈·PR
- 주요 파일 또는 문서
- 검증 상태
- 남은 작업과 리스크

## 5. 세션 운영 규칙

### 5.1 기존 세션을 계속 사용하는 조건

다음 조건을 대체로 만족하면 같은 세션을 유지한다.

- 세션의 결과물과 완료 조건이 바뀌지 않았다.
- 같은 파일군과 기술적 개념을 계속 다룬다.
- 이전 결정과 시행착오가 다음 작업에 직접 필요하다.
- 현재 세션이 목표, 현재 상태, 주요 결정, 검증 결과와 다음 작업을 정확히 설명할 수 있다.
- 테스트 실패 분석과 수정처럼 하나의 검증 루프 안에서 작업하고 있다.

세션이 여러 날 지속되거나 컨텍스트가 압축됐다는 이유만으로 새 세션을 만들지 않는다.

### 5.2 새 세션을 시작하는 조건

다음 중 하나에 해당하면 새 세션을 시작한다.

- 만들려는 최종 결과물이 달라졌다.
- 같은 업무 항목 안에서도 설계, 구현, 테스트, 운영 대응 등 독립적으로 검토할 결과물이 생겼다.
- 작업 파일군이나 기술적 관점이 크게 바뀌었다.
- 기존 세션의 전제와 현재 코드 상태가 달라졌다.
- 목표 또는 완료 조건이 크게 변경됐다.
- 에이전트가 이미 확정된 결정을 반복해서 묻거나 모순된 판단을 한다.
- 세션 요약만으로 현재 상태를 신뢰하기 어렵다.
- 다른 작업과 병렬로 독립적으로 진행할 수 있다.

간단한 판단식은 다음과 같다.

```text
같은 업무 항목 + 같은 결과물 + 같은 완료 조건 = 기존 세션 계속
같은 업무 항목 + 새로운 결과물 = 새 세션 + 인수인계 문서
다른 업무 항목 = 새 세션
```

### 5.3 직접 연관된 수정

세션 목표를 달성하기 위해 필요한 작은 연관 수정은 같은 세션에서 수행할 수 있다.

예를 들어 테스트 세션에서 테스트를 통과시키기 위해 작은 구현 오류를 수정하는 것은 같은 검증 결과물에 속한다. 반면 테스트 과정에서 전체 인증 구조를 재설계해야 한다면 새 결과물로 보고 별도 세션을 시작한다.

## 6. 긴 세션과 컨텍스트 압축

컨텍스트 압축은 세션 종료 조건이 아니다. 압축 이후에도 다음 질문에 정확히 답할 수 있다면 기존 세션을 유지할 수 있다.

- 현재 목표는 무엇인가?
- 완료 조건은 무엇인가?
- 지금까지 무엇을 완료했는가?
- 어떤 결정을 내렸고 그 이유는 무엇인가?
- 무엇을 검증했고 어떤 검증이 남았는가?
- 다음 작업은 무엇인가?

답이 불분명하거나 현재 코드와 일치하지 않는다면 다음 순서로 전환한다.

1. 현재 단계 체크포인트 작성
2. 업무 항목의 `context.md` 갱신
3. 새 세션 시작
4. 새 세션에서 `context.md`와 실제 코드·Git 상태 확인
5. 새 세션의 결과물과 완료 조건 선언

## 7. 인수인계 문서

각 활성 업무 항목은 다음 위치에 현재 상태 문서를 가진다.

```text
work-items/<work_item_id>/context.md
```

`context.md`는 전체 세션 transcript의 요약본이 아니다. 새 세션이 빠르고 정확하게 작업을 시작하기 위한 짧은 안내서다.

권장 구성:

```markdown
# <업무 항목 ID와 제목>

## 목표
## 현재 상태
## 주요 결정과 이유
## 주요 파일과 문서
## 검증 상태
## 남은 작업
## 리스크와 확인할 사항
## 마지막으로 확인한 Git 기준점
```

운영 원칙:

- `context.md`는 현재 상태를 반영하도록 갱신할 수 있다.
- 과거 작업 이력은 `context.md`에 계속 누적하지 않는다.
- 소스 코드와 Git 상태가 최종 사실이며, 새 세션은 문서만 신뢰하지 않고 실제 상태를 다시 확인한다.
- 프로젝트 전체에 항상 적용되는 규칙은 `context.md`가 아니라 `AGENTS.md` 또는 프로젝트 문서에 둔다.

## 8. 체크포인트 기록

### 8.1 기록 시점

다음 상황에서는 체크포인트를 작성한다.

1. 업무일 종료 시
2. 의미 있는 단계 또는 검증이 완료됐을 때
3. 다른 결과물이나 업무 항목으로 전환하기 전
4. 새 세션으로 넘기거나 현재 세션을 종료하기 전
5. 긴 중단 전에 현재 상태를 보존해야 할 때
6. 컨텍스트 압축 전후로 맥락 신뢰도를 보강해야 할 때

매 응답이나 모든 도구 호출마다 의미 기록을 생성하지 않는다.

### 8.2 기록 범위

사용자가 별도 범위를 지정하지 않으면 체크포인트는 **현재 업무 항목의 마지막 체크포인트 이후부터 지금까지**의 변경분만 기록한다.

체크포인트 작성 절차:

1. 현재 프로젝트, 업무 항목과 세션을 식별한다.
2. 이전 체크포인트와 현재 Git 기준점을 확인한다.
3. 해당 구간의 작업 목적, 변경, 결정, 검증과 다음 작업을 정리한다.
4. 근거가 없는 결과나 성과는 사실로 단정하지 않는다.
5. 새 체크포인트 파일을 생성한다.
6. 필요하면 `context.md`를 현재 상태로 갱신한다.
7. 생성된 기록 경로와 핵심 내용을 사용자에게 알린다.

### 8.3 불변성

- 체크포인트는 append-only 기록이다.
- 사실 오류를 정정해야 하면 기존 기록을 덮어쓰지 않고 정정 체크포인트를 추가한다.
- 보고서나 인수인계 문서는 파생 산출물이므로 다시 생성하거나 갱신할 수 있다.

## 9. 날짜와 시간

### 9.1 기본 정책

세션은 여러 날 지속될 수 있지만 체크포인트의 정상적인 기록 범위는 날짜를 넘기지 않는다.

같은 세션을 다음 날에도 계속할 경우 세션을 닫지 않고 업무일 종료 시 다음과 같이 요청한다.

> 오늘 작업은 여기까지 기록해줘. 세션은 계속 유지한다.

### 9.2 시간 필드

기록은 작업 시각과 기록 시각을 구분한다.

```yaml
captured_at: 2026-07-13T18:10:00+09:00
work_period:
  start: 2026-07-13T09:30:00+09:00
  end: 2026-07-13T18:00:00+09:00
  precision: exact
  basis:
    - checkpoint
```

- `captured_at`: 기록 파일을 실제로 생성한 정확한 시각
- `work_period.start`, `work_period.end`: 기록이 다루는 실제 작업 구간
- `precision`: 날짜·시간 정확도
- `basis`: 날짜를 판단한 근거

허용하는 정확도:

| 값 | 의미 |
| --- | --- |
| `exact` | 시작과 종료 시각에 신뢰 가능한 근거가 있음 |
| `day` | 작업 날짜는 확실하지만 시각은 확실하지 않음 |
| `range` | 여러 날짜 범위만 확실함 |
| `unknown` | 실제 작업 날짜를 신뢰성 있게 판단할 수 없음 |

허용하는 주요 근거:

- `checkpoint`: 당일 명시적 기록
- `activity-event`: 자동 활동 이벤트의 타임스탬프
- `transcript`: 에이전트 제품이 제공한 transcript 메타데이터
- `git`: 커밋 등 Git 이력
- `user`: 사용자가 직접 지정한 날짜

### 9.3 날짜별 폴더

체크포인트 파일은 `captured_at` 날짜를 기준으로 저장한다.

```text
records/YYYY/MM/DD/<checkpoint_id>.md
```

정상적인 당일 기록에서는 저장 날짜와 작업 날짜가 같다. 과거 작업을 뒤늦게 기록한 경우 실제 작업 날짜는 `work_period`가 표현하며, 보고서 생성 시 폴더 경로가 아니라 메타데이터를 기준으로 집계한다.

## 10. 장기 세션의 날짜 분류

일주일 동안 한 세션을 유지하는 것은 허용한다. 다만 마지막 날 한 번만 기록을 요청하면 에이전트가 이전 작업을 날짜별로 정확히 복원할 수 있다고 가정하지 않는다.

권장 방식:

```text
하나의 세션
├── 7월 7일 체크포인트
├── 7월 8일 체크포인트
├── 7월 9일 체크포인트
└── 7월 10일 체크포인트
```

각 기록은 같은 `work_item_id`와 가능하면 같은 `source_session_ref`를 가진다. 보고서는 이를 하나의 업무 진행으로 합친다.

## 11. 누락 기록과 사후 복원

여러 날 동안 체크포인트를 남기지 않았다면 다음 근거 순서로 작업 기간을 복원한다.

1. 신뢰 가능한 transcript 타임스탬프
2. 자동 활동 이벤트
3. Git 커밋과 테스트·빌드 로그
4. 사용자가 지정한 작업 날짜
5. 현재 세션의 의미 기반 요약

복원 규칙:

- 근거가 없는 일별 분할은 금지한다.
- 날짜가 확실하지 않으면 `precision: range` 또는 `unknown`을 사용한다.
- 커밋 날짜가 실제 모든 작업의 날짜를 의미한다고 가정하지 않는다.
- 추정 결과는 확정 사실과 구분한다.
- 뒤늦게 생성한 기록의 `captured_at`을 과거 날짜로 위조하지 않는다.

## 12. 자동 활동 이벤트

자동 활동 이벤트는 날짜와 실행 근거를 보완하는 선택적 보조 계층이다. 의미 기반 체크포인트를 대체하지 않는다.

수집 가능한 항목:

- 타임스탬프
- 에이전트와 세션 참조
- 프로젝트·저장소·브랜치
- Git HEAD
- 변경된 파일 이름
- 테스트·빌드 실행과 결과
- 커밋·PR·이슈 참조

기본적으로 저장하지 않는 항목:

- 전체 세션 transcript
- 전체 프롬프트
- 전체 소스 diff
- 환경 변수와 비밀값
- 명령 출력에 포함된 민감 데이터

자동 이벤트는 체크포인트의 근거로 참조할 수 있지만, 이벤트만으로 업무 목적이나 성과를 임의로 생성하지 않는다.

## 13. 표준 사용자 요청

에이전트 간 동작을 일관되게 하기 위해 다음 표현을 표준 의도로 취급한다.

### 당일 기록

> 오늘 작업은 여기까지 기록해줘.

현재 날짜의 마지막 체크포인트 이후 작업을 기록한다. 세션은 계속 유지할 수 있다.

### 단계 기록

> 현재 단계까지 체크포인트를 기록해줘.

의미 있는 단계 완료 지점까지 기록한다.

### 새 세션 인수인계

> 새 세션에서 이어갈 수 있도록 인수인계 문서를 갱신해줘.

체크포인트를 작성하고 현재 업무 항목의 `context.md`를 갱신한다.

### 업무 완료

> 이 업무를 완료 처리하고 최종 기록을 작성해줘.

최종 검증과 결과를 기록하고 업무 항목 상태를 완료로 변경한다. 검증되지 않은 항목이 있으면 완료로 단정하지 않고 남은 위험을 명시한다.

### 범위 지정 기록

> 지난 체크포인트 이후 작업을 기록해줘.

명시한 체크포인트 이후의 변경분만 기록한다.

## 14. 보고서 생성 원칙

- 보고서는 지정 기간에 해당하는 체크포인트를 읽는다.
- 기록은 `work_item_id`를 기준으로 묶고 세션별 나열을 기본 형식으로 사용하지 않는다.
- 동일 작업의 중복 체크포인트를 단순 합산하지 않는다.
- 활동, 결과, 영향과 성과를 구분한다.
- 모든 중요한 결과와 성과 문장은 체크포인트, Git, PR, 이슈 또는 검증 결과를 근거로 가진다.
- 수치적 영향이 확인되지 않았으면 수치를 추정하지 않는다.
- 진행 중, 완료, 보류, 차단 상태를 구분한다.
- 실제 출력 순서와 문체는 사용자가 제공하는 보고 양식 템플릿을 따른다.

## 15. 에이전트 책임

기록을 작성하는 에이전트는 다음을 지켜야 한다.

- 현재 세션 전체가 아니라 요청된 체크포인트 범위를 기록한다.
- 세션 대화와 실제 저장소 상태가 다르면 저장소 상태를 우선 확인한다.
- 실행하지 않은 테스트를 통과했다고 기록하지 않는다.
- 작업 활동을 업무 성과로 과장하지 않는다.
- 날짜와 근거의 정확도를 명시한다.
- 민감 정보와 원문 transcript를 기본 기록에 포함하지 않는다.
- 기존 기록과 업무 항목을 검색하여 중복 생성을 줄인다.
- 새 세션 전환 시 필요한 현재 상태만 인수인계 문서에 남긴다.

## 16. 예시 시나리오

### 16.1 테스트 작업이 사흘 동안 이어지는 경우

- 결과물: 인증 테스트 스위트 완성
- 세션: 하나를 유지
- 체크포인트: 매일 하나 이상 생성
- 업무 항목: `AUTH-142` 유지
- 보고서: 세 날짜 기록을 인증 개선 업무로 통합

### 16.2 테스트 중 아키텍처 재설계가 필요한 경우

- 기존 테스트 세션에서 발견 사항을 체크포인트로 기록
- `context.md`에 재설계가 필요한 이유와 관련 파일을 갱신
- 아키텍처 재설계를 별도 결과물로 정의하고 새 세션 시작
- 재설계 완료 후 테스트 세션을 재개하거나 새로운 테스트 세션 시작

### 16.3 일주일 뒤 처음 기록하는 경우

- transcript, 활동 이벤트와 Git 이력을 확인
- 확실한 날짜만 일별로 분류
- 나머지는 여러 날짜 범위 기록으로 보존
- 날짜별 작업을 추측하여 채우지 않음

## 17. 구현 기준

향후 Work Harness의 CLI, Codex Skill, Claude Code 명령과 Hook은 이 문서의 의미를 동일하게 구현해야 한다.

구현 우선순위:

1. 업무 항목과 체크포인트 스키마
2. `context.md` 템플릿
3. 명시적 체크포인트 생성
4. 날짜 정확도와 근거 처리
5. 주간 보고서 집계
6. 자동 활동 이벤트와 누락 감지

구현 세부사항이 이 문서와 충돌하면 구현을 수정하거나, 운영 원칙을 변경해야 할 충분한 이유가 있을 때 이 문서의 버전을 올려 변경 내용을 기록한다.
